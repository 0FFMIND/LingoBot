# 记忆管理模块当前运作方式

本文档按当前代码状态梳理：哪些内容进 Redis 短期缓存，哪些内容进 PostgreSQL 长期记录，以及 `learning/memory` 模块现在实际有没有接入主流程。

## 总览

当前项目里的“记忆”大致分成三层：

1. 内存对象层：`com.lingobot.learning.memory.core`
   - `ShortTermMemory`、`LongTermMemory`、`MemoryItem` 等是纯 Java 内存结构。
   - 目前没有看到它们被 Spring Bean 持久化，也没有接 Redis/PostgreSQL。
   - 换句话说，这一层更像是预留的抽象模型，目前不是线上数据入口。

2. Redis 短期缓存层
   - 主要用于“当前学习状态”“词汇卡快照”“最近卡片/事件缓存”“词汇卡查询缓存”。
   - 这些数据有 TTL，偏临时、加速读取、辅助前端轮询或避免懒加载问题。

3. PostgreSQL 长期记忆层
   - 主要是 `user_vocabularies`、`vocabulary_cards`、`user_vocabulary_events`。
   - 用来保存用户长期学习进度、历史词汇卡、学习事件。

## ShortTermMemory 当前不是 Redis

`ShortTermMemory` 当前实现是一个内存队列：

- 文件：`lingobot-backend/src/main/java/com/lingobot/learning/memory/core/ShortTermMemory.java`
- 最大容量：`MAX_ITEMS = 50`
- 超过 50 条时丢弃最早的一条。
- 存储结构：`Deque<MemoryItem>`
- 没有 Redis key。
- 没有数据库表。
- 当前代码中没有看到它被业务服务调用。

所以严格来说：当前“短期记忆”这个类还没有真正进入 Redis，它只是一个本地内存容器。

## Redis 里实际存了什么

### 1. 当前词汇学习状态

服务：`VocabularyStateService`

Redis key：

```text
vocabulary:state:{conversationId}
```

写入时机：

- AI 工具 `display_flashcard` 被调用时，`VocabularyTool` 调用 `vocabularyStateService.saveCurrentWord(...)`。

内容：

- `word`
- `phonetic`
- `partOfSpeech`
- `meaning`
- `synonyms`
- `vocabularyCategory`
- `vocabularyDifficulty`

TTL：

```text
24 hours
```

用途：

- 保存当前对话正在学习的词汇状态。
- 更像“当前会话状态缓存”，不是长期学习记忆。

### 2. 词汇卡查询缓存

服务：`VocabularyCardServiceImpl`

Redis key：

```text
vocabulary:card:{cardId}
vocabulary:cards:{conversationId}
vocabulary:count:{conversationId}
```

内容：

- 单张词汇卡 DTO
- 某个 conversation 的有效词汇卡列表
- 某个 conversation 的有效词汇卡数量

TTL：

```text
1 hour
```

失效时机：

- 创建词汇卡
- 更新用户释义
- 更新用户造句
- 更新释义检查/句子分析结果
- 标记完成
- 重新生成
- 删除卡片

用途：

- 减少数据库查询。
- 保证前端轮询时可以在异步检查完成后看到最新状态。

### 3. 词汇卡快照缓存

服务：`VocabularyCardSnapshotServiceImpl`

Redis key：

```text
vocabulary:snapshot:card:{userId}:{cardId}
vocabulary:snapshot:recent_ids:{userId}
```

内容：

- `VocabularyCardSnapshot`
- 最近卡片 ID 列表

TTL：

```text
7 days
```

写入时机：

- `saveCardSnapshot(userId, card)`
- `markCardAsRecent(userId, cardId)`

用途：

- 给用户维度保存词汇卡快照。
- 避免后续读取时触发 JPA lazy loading。
- 快速取最近学习过的卡片。

注意：

- 快照是 Redis 缓存，不是权威长期数据源。
- Redis miss 时会回落到 PostgreSQL 的 `vocabulary_cards` 重新构建快照。

### 4. L1 短时词汇记忆

服务：`VocabularyMemoryServiceImpl`

Redis key：

```text
memory:l1:recent:{userId}
memory:l1:wrong:{userId}
memory:l1:regenerated:{userId}
```

事件类型：

```java
SEEN
CORRECT
WRONG
REGENERATED
UNKNOWN
```

#### `memory:l1:recent:{userId}`

含义：

- 最近 10 个接触过的词。
- “接触”包括生成、看到、答对、答错、重新生成等互动。
- 最新记录放在最前面。
- 按 word 去重，重复接触同一个词会刷新到列表头部。

TTL：

```text
7 days
```

作用：

- 短期去重。
- 避免用户短时间内反复遇到同一个词。

#### `memory:l1:wrong:{userId}`

含义：

- 最近 10 个答错词。
- 答错来源包括释义检查或句子分析整体失败。

TTL：

```text
14 days
```

作用：

- 错误比正确更值得短期保留。
- 用于复习优先级。
- TTL 不短于 recent。

#### `memory:l1:regenerated:{userId}`

含义：

- 最近 10 个被用户重新生成过的词。
- 记录的是被替换掉的旧词卡。
- `REGENERATED` 是单独保留的不满意信号。

TTL：

```text
14 days
```

作用：

- 生成多样性。
- 避免继续生成用户刚刚不满意的词。
- 给 prompt 提供“用户不喜欢/不需要这类词”的短期信号。

#### 当前会话 recent cards

来源：

- PostgreSQL `vocabulary_cards`
- `vocabularyCardRepository.findByConversationIdOrderByPositionAsc(conversationId)`

含义：

- 当前 conversation 内最近/历史全部词卡。
- 包含已重新生成的旧卡。
- 已重新生成的旧卡在 `VocabularyMemoryRecord.eventType` 中标记为 `REGENERATED`。

作用：

- 当前会话避免重复。
- 保留“重新生成”这个强信号。

### 5. 学习事件缓存

服务：`UserVocabularyEventServiceImpl`

Redis key：

```text
vocabulary:events:{userId}:card:{vocabularyCardId}
vocabulary:recent_cards:{userId}
```

内容：

- 某张卡片的学习事件列表
- 用户最近交互过的卡片 ID 列表

TTL：

```text
7 days
```

写入时机：

- `recordMeaningCheckEvent(...)`
- `recordSentenceAnalysisEvent(...)`
- `recordSeenEvent(...)`
- `recordCardCompletedEvent(...)`

用途：

- 加速按卡片读取学习事件。
- 维护最近卡片列表。

注意：

- 事件首先写 PostgreSQL，然后同步缓存到 Redis。
- Redis 是加速层，PostgreSQL 才是长期事件记录。

## PostgreSQL 里实际存了什么

### 1. `vocabulary_cards`

实体：`VocabularyCard`

保存内容：

- 单词、音标、词性、释义、例句、例句翻译
- 用户释义猜测
- 释义检查结果
- 用户英文造句
- 句子分析结果
- 是否完成
- 是否被重新生成
- conversation 归属
- vocabulary word id

写入时机：

- 创建词汇卡
- AI 生成下一张卡
- 重新生成卡片
- 用户提交释义/造句
- 异步释义检查或句子分析回写结果
- 标记完成

这张表是“词汇卡历史”的长期来源。

### 2. `user_vocabularies`

实体：`UserVocabulary`

保存内容：

- userId
- vocabularyWordId
- 展示字段：word、phonetic、partOfSpeech、meaning、synonyms、category、difficulty
- 学习状态：`NEW / LEARNING / REVIEWING / MASTERED / IGNORED`
- 掌握度：`masteryScore`
- seen/correct/wrong 计数
- firstSeenAt、lastSeenAt、nextReviewAt
- lastEventType

写入时机：

- `upsertRecordOnly(userId, card)`
  - 只同步词汇展示字段。
  - 不增加学习次数。
  - 不改变学习状态。

- `upsertProgress(userId, card)` / `upsertProgress(userId, vocabularyWordId)`
  - 增加 seen count。
  - 设置 lastSeenAt。
  - NEW 会变成 LEARNING。
  - 初始掌握度至少设到 0.12。
  - 设置 nextReviewAt。

- `updateProgress(userId, vocabularyWordId, isCorrect)`
  - 根据测验结果更新 correct/wrong count。
  - 重新计算 masteryScore。
  - 根据掌握度切换 LEARNING / REVIEWING / MASTERED。
  - 重新调度 nextReviewAt。

这张表是“用户对某个标准化单词的长期学习状态”。

### 3. `user_vocabulary_events`

实体：`UserVocabularyEvent`

保存内容：

- userId
- vocabularyWordId
- userVocabularyId
- vocabularyCardId
- eventType
- 是否正确
- 掌握度变化
- meaning check 的用户答案和反馈
- sentence analysis 的用户答案和反馈
- createdAt

写入时机：

- 释义检查事件
- 句子分析事件
- seen 事件
- 卡片完成事件

这张表是“学习行为流水”，适合以后做复盘、统计、推荐和记忆检索。

## VocabularyMemoryService 当前怎么取记忆

服务：`VocabularyMemoryServiceImpl`

入口：

```java
retrieveMemory(userId, conversationId, intent, constraints)
```

它会从 PostgreSQL 读两类数据：

1. 当前对话已出现过的词
   - 来源：`vocabularyCardRepository.findByConversationIdOrderByPositionAsc(conversationId)`
   - 结果：`excludedWords`
   - 作用：避免重复生成同一个对话里已经出现过的词。

2. 用户长期词汇记录
   - 来源：`userVocabularyRepository.findByUserId(userId)`
   - 分成三类：
     - `recentWords`：按 `updatedAt` 倒序，最多 20 个
     - `masteredWords`：`masteryScore >= 0.8`，最多 30 个
     - `weakWords`：`masteryScore < 0.4`，最多 15 个

然后 `VocabularyMemoryPromptBuilder` 会把这些内容拼成 prompt 片段：

```text
## Vocabulary Learning Memory

### Recently Learned Words
...

### Mastered Words
...

### Words Needing Review
...

### Words to Exclude (already used recently)
...
```

## 当前实际接入情况

当前代码里，`VocabularyMemoryServiceImpl` 和 `VocabularyMemoryPromptBuilder` 已经存在，也在 `VocabularyCardServiceImpl` 里声明了依赖字段。

但从当前 `VocabularyCardServiceImpl.generateRandomWord(...)` 看到的实际逻辑是：

1. 读取用户偏好。
2. 生成基础 system prompt。
3. 调用 `buildVocabularyHistoryForPrompt(conversationId)`。
4. 把当前 conversation 的词汇卡历史拼进 prompt。
5. 调用 `ToolLoopService.executeOneTimeToolCall(...)` 让 AI 生成词汇卡。

也就是说，当前实际进入 AI 词汇生成 prompt 的主要是：

- 当前 conversation 的历史词汇卡
- 不是 `VocabularyMemoryService.retrieveMemory(...)` 生成的完整用户长期记忆上下文

因此目前状态可以理解为：

- PostgreSQL 长期记忆数据已经在积累。
- Redis 短期缓存和快照也在工作。
- `VocabularyMemoryService` 的长期记忆检索能力已经写好。
- 但词汇生成主流程目前还没有真正切到 `VocabularyMemoryService + VocabularyMemoryPromptBuilder` 这条链路。

## 简单判断规则

会进 Redis 的内容：

- `memory:l1:recent:{userId}`：最近 10 个接触过的词，TTL 7 天。
- `memory:l1:wrong:{userId}`：最近 10 个答错词，TTL 14 天。
- `memory:l1:regenerated:{userId}`：最近 10 个重新生成词，TTL 14 天。
- 当前对话正在学习的词汇状态。
- 词汇卡 DTO/list/count 查询缓存。
- 用户最近词汇卡快照。
- 用户学习事件的短期缓存。
- 最近卡片 ID 列表。

会进 PostgreSQL 的内容：

- 词汇卡本体和历史。
- 用户每个单词的长期学习状态。
- 用户每次学习行为事件。
- 掌握度、复习时间、答题结果等长期学习数据。

暂时不会自动进任何持久化的内容：

- `ShortTermMemory` / `LongTermMemory` 这套 core 类里的 `MemoryItem`。
- 除非后续有服务显式把它们接入 Redis 或数据库。
