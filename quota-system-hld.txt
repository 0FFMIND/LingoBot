LingoBot · 额度与消费系统  High-Level Design
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━


1. 问题与目标
─────────────────────────────────────────────────────────────────────

用户调用付费 AI 功能（对话、语音、图片）时，系统需要保证：

  · 余额不足时拒绝调用，不透支
  · 调用成功后按实际消耗扣费
  · 并发请求下不超扣、不漏扣
  · 故障或调用失败时不错误扣费
  · 消费记录可追溯

3. 核心设计决策
─────────────────────────────────────────────────────────────────────

决策一：Redis 做快速通道，PostgreSQL 做真相来源

  原因：每次 AI 调用都需要读余额，若直接打 DB 在高并发下压力大。
  方案：Redis 缓存余额快照（短 TTL），读多写少的场景下大幅降低 DB 读。
        真正扣费时必须写 PostgreSQL，Redis 仅是加速层，不是权威数据。
  取舍：引入缓存一致性问题，通过"扣费后主动刷新缓存 + TTL 兜底"解决。


决策二：冻结机制防并发超扣

  原因：用户同时发起两个 AI 请求，两个请求都读到同一余额，可能都通过校验，
        最终合计扣费超出余额。
  方案：调用前在 Redis 中"冻结"预估费用，可用余额 = 实际余额 - 已冻结。
        调用结束后按实际用量扣费并释放冻结。
  取舍：若 Redis 不可用，降级为直接走 DB 悲观锁路径，性能下降但结果正确。


决策三：DB 层悲观锁作为最后防线

  原因：Redis 是缓存，存在极端情况下的数据不一致窗口。
  方案：扣费写入 DB 时对用户行加锁（SELECT FOR UPDATE），串行化扣费操作，
        在 DB 层做最终的余额充足性校验。
  取舍：锁粒度是用户级别，同一用户的并发扣费串行执行，延迟略增，但强一致。


决策四：流式调用（SSE）与普通调用统一扣费路径

  原因：流式 AI 调用无法在开始前知道输出 token 数，无法精确预扣。
  方案：流式调用同样冻结预估费用，在流结束的回调钩子中读取实际 token 用量
        并完成扣费。流中断（超时/断连）时按已产生内容计费，不全额退款。
  取舍：实现复杂度高于普通调用，但保证了用户和系统之间的费用公平性。


4. 数据流（一次 AI 调用的生命周期）
─────────────────────────────────────────────────────────────────────

  请求进入
      │
      ├─[1] 读缓存余额 → 余额不足 → 直接拒绝（快速失败，不打 LLM）
      │
      ├─[2] 冻结预估费用（Redis 原子操作）
      │
      ├─[3] 调用 LLM → 获得实际 token 用量
      │
      ├─[4] 写 PostgreSQL：悲观锁 + 扣费 + 记录消费流水
      │
      └─[5] 刷新 Redis 缓存，释放冻结额度

  异常路径（任意步骤失败）：
      ├─[3] LLM 调用失败 → 释放冻结，不扣费
      └─[4] DB 写入失败 → 事务回滚，余额不变，Redis 等待 TTL 过期自动修正


5. 存储职责划分
─────────────────────────────────────────────────────────────────────

  PostgreSQL                        Redis
  ──────────────────────────────    ──────────────────────────────
  · 用户余额（权威）                · 余额快照（缓存，短期有效）
  · 每笔消费流水（永久保留）        · 当前冻结金额（调用期间临时）
  · 充值/退款记录                   · 调用频率计数（限流）
  · 幂等记录（防重复扣费）

  原则：PostgreSQL 存的是"发生了什么"，Redis 存的是"现在大概是什么状态"。


6. 故障场景与应对
─────────────────────────────────────────────────────────────────────

  故障               影响                  应对
  ─────────────────────────────────────────────────────────────────
  Redis 宕机         余额读取变慢          降级：直接查 DB，跳过冻结步骤
  DB 扣费超时        本次调用失败          事务回滚，冻结自动过期后释放
  LLM 无响应         流式连接超时          onTimeout 回调：按已生成量计费
  客户端断连         SSE 连接中断          onError 回调：释放冻结，不扣费
  重复请求（重试）   可能重复扣费          幂等键：相同请求 ID 只扣一次


7. 未覆盖的部分（意识到但不在本期范围）
─────────────────────────────────────────────────────────────────────

  · 充值入口（支付网关对接）
  · 实时余额推送到前端
  · 按月/按天的用量统计报表
  · 不同用户等级的差异化定价
