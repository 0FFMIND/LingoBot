# Chat 模块

## 模块概述

Chat 模块是聊天机器人的核心业务模块，负责处理用户与 AI 的对话交互。该模块协调 LLM 模型调用、MCP 工具执行、SSE 流式输出等功能，实现了完整的聊天体验，包括普通聊天模式和 Agent 智能体模式。

## 主要功能

- **对话处理**：接收用户消息，调用 AI 模型生成响应
- **流式输出**：通过 SSE 实时推送 AI 响应片段
- **工具调用循环**：支持多轮工具调用，实现复杂任务
- **多模式支持**：普通聊天模式和 Agent 智能体模式
- **多模态输入**：支持语音输入（音频消息）
- **消息历史**：构建对话上下文，支持多轮对话

## 模块结构

### controller/ 控制器层

#### `ChatController` 聊天控制器
- 发送消息（非流式）：`POST /api/chat`
- 发送消息（流式）：`POST /api/chat/stream`
- 获取对话消息：`GET /api/chat/{conversationId}/messages`
- 重试消息：`POST /api/chat/retry`
- 编辑消息：`PUT /api/chat/edit`

### service/ 服务层

#### `ChatService` 聊天服务接口
- 定义聊天相关的核心方法

#### `ChatServiceImpl` 聊天服务实现
- 处理用户消息发送
- 协调工具调用和模型响应
- 管理对话消息的存储

#### `MessageHistoryService` 消息历史服务
- 构建对话上下文
- 从数据库加载历史消息
- 转换为模型可识别的格式
- 支持截断到指定消息索引

#### `ToolLoopService` 工具循环服务
- 执行多轮工具调用
- 解析模型的工具调用请求
- 调用 MCP 工具执行
- 处理工具结果并继续对话
- 限制最大工具调用次数

#### `SseEmitterService` SSE 推送服务
- 创建和管理 SSE 连接
- 流式推送 AI 响应
- 支持 Agent 模式的工具调用事件
- 管理连接生命周期（超时、错误、完成）
- 推送多种事件类型（思考中、工具调用、工具结果、内容、完成）

### dto/ 数据传输对象

#### `ChatRequest` 聊天请求
- 对话 ID、消息内容
- 模式选择（chat/agent）
- 模型选择
- 音频消息支持（音频数据、格式、时长）

#### `EditMessageRequest` 编辑消息请求
- 对话 ID、用户消息 ID
- 新消息内容

#### `StreamEvent` 流式事件
- 事件类型（thinking、tool_call、tool_result、content、done、error）
- 事件数据
- 用于 SSE 流式输出

## 聊天模式

### 普通聊天模式 (Chat Mode)
- 单轮工具调用
- 工具执行后直接返回结果
- 适用于简单查询和任务

### Agent 智能体模式 (Agent Mode)
- 多轮工具调用循环
- 支持复杂推理和规划
- 实时推送思考和工具调用事件
- 用户可以观察 AI 的推理过程
- 适用于复杂任务（如代码分析、多步骤操作）

## 工具调用流程

### 1. 检测工具调用
- 模型响应中检测工具调用标记
- 解析工具名称和参数

### 2. 执行工具
- 通过 `ToolLoopService` 调用 `McpService`
- 执行相应的 MCP 工具
- 获取工具执行结果

### 3. 处理结果
- 将工具结果添加到对话上下文
- 再次调用模型获取响应
- 循环直到模型返回文本响应

### 4. 事件推送（Agent 模式）
- 推送 `thinking` 事件：AI 正在思考
- 推送 `tool_call` 事件：调用工具
- 推送 `tool_result` 事件：工具执行结果
- 推送 `content` 事件：最终响应内容
- 推送 `done` 事件：对话完成

## 流式响应流程

### 1. 创建 SSE 连接
- 客户端请求流式接口
- 服务端创建 `SseEmitter`
- 设置超时和回调

### 2. 处理消息
- 根据模式选择处理方式
- Agent 模式：执行完整工具循环
- 普通模式：单次工具调用检查

### 3. 推送响应
- 文本响应：逐字推送 `content` 事件
- 工具事件：推送 `tool_call` 和 `tool_result` 事件
- 完成事件：推送 `done` 事件，包含最终消息

### 4. 连接管理
- 超时自动关闭
- 错误处理和通知
- 客户端断开时清理资源

## 多模态支持

### 音频消息处理
- 支持语音输入
- 音频数据 Base64 编码
- 自动格式转换（通过 Audio 模块）
- 使用支持音频的模型（如 Qwen Omni）

### 音频消息流程
1. 前端录制语音，转换为 Base64
2. 发送包含音频数据的聊天请求
3. 后端检测音频消息标志
4. 调用支持音频的模型
5. 返回文本响应

## 消息重试与编辑

### 重试消息
- 重新处理用户的最后一条消息
- 从数据库加载历史消息
- 重新调用 AI 模型
- 生成新的响应

### 编辑消息
- 修改历史消息内容
- 删除该消息之后的所有消息
- 重新处理编辑后的消息
- 生成新的对话分支

## 依赖关系

Chat 模块依赖以下模块：
- **LLM 模块**：调用 AI 模型生成响应
- **MCP 模块**：执行工具调用
- **Conversation 模块**：管理对话和消息存储
- **Audio 模块**：处理音频格式转换
- **Common 模块**：异常处理、通用响应
