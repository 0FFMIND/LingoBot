一、问题背景与现状分析
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【现有单词卡流程】
  用户开一个"词汇扩展"对话 → AI 展示单词卡（VocabularyCard）
  → 用户猜测词义、造句 → AI 检测反馈 → 对话结束

【核心问题】
  1. 单词卡与 Conversation 强绑定（VocabularyCard.conversationId），
     对话结束后学习记录完全孤立，没有跨会话的记忆。

  2. 没有遗忘曲线/间隔重复机制，学过的单词不会被合理安排复习。

  3. AI 不知道用户已经学过哪些单词，可能反复推送已掌握的词汇。

  4. 用户无法查看自己的"词汇本"，不知道自己学了多少单词。

【目标】
  - 建立跨会话的单词学习历史（Word Memory）
  - 引入间隔重复算法（SRS）驱动复习
  - 让 AI 具备"用户词汇记忆"上下文意识
  - 提供词汇统计与回顾界面

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【3.1 核心表：word_memory】

  CREATE TABLE word_memory (
    id                BIGSERIAL PRIMARY KEY,
    user_id           BIGINT NOT NULL REFERENCES users(id),
    word              VARCHAR(100) NOT NULL,        -- 标准化（小写）
    phonetic          VARCHAR(100),                 -- 音标
    meaning           TEXT NOT NULL,                -- 词义（首次学习时存储）
    example           TEXT,                         -- 例句

    -- 学习指标
    first_seen_at     TIMESTAMP NOT NULL,
    last_reviewed_at  TIMESTAMP,
    next_review_at    TIMESTAMP,                    -- SRS 调度时间
    review_count      INT DEFAULT 0,
    correct_count     INT DEFAULT 0,
    incorrect_count   INT DEFAULT 0,

    -- SM-2 算法参数
    ease_factor       FLOAT DEFAULT 2.5,            -- 难易系数（初始 2.5）
    interval_days     INT DEFAULT 1,                -- 下次复习间隔（天）

    -- 掌握度状态机
    -- LEARNING（新学）→ REVIEWING（复习中）→ MASTERED（已掌握）→ FORGOTTEN（遗忘，需重学）
    mastery_status    VARCHAR(20) DEFAULT 'LEARNING',

    -- 溯源信息（用于"这个词我是在哪学的"）
    source_conversation_ids  TEXT,                  -- JSON 数组，逗号分隔的 conversationId

    -- 用户造句样本（最多保存 3 句，JSON 数组）
    user_sentences    TEXT,

    UNIQUE(user_id, word)
  );

  CREATE INDEX idx_word_memory_user_next_review
    ON word_memory(user_id, next_review_at);

  CREATE INDEX idx_word_memory_user_status
    ON word_memory(user_id, mastery_status);


【3.2 复习记录表：word_review_log】
  用于详细追踪每次复习行为，方便统计和回溯。

  CREATE TABLE word_review_log (
    id                BIGSERIAL PRIMARY KEY,
    word_memory_id    BIGINT NOT NULL REFERENCES word_memory(id),
    user_id           BIGINT NOT NULL,
    reviewed_at       TIMESTAMP NOT NULL DEFAULT NOW(),
    review_type       VARCHAR(20),                  -- MEANING_GUESS / SENTENCE / FLASHCARD
    is_correct        BOOLEAN,
    response_time_ms  INT,                          -- 用户作答响应时间（毫秒）
    quality_score     INT                           -- SM-2 的 q 值（0~5）
  );

  CREATE INDEX idx_review_log_user_date
    ON word_review_log(user_id, reviewed_at);


【3.3 与现有表的关系】

  VocabularyCard（已有）
  ├── conversationId  → Conversation
  ├── word, meaning, phonetic, example...
  └── meaningIsCorrect, userSentence, aiFeedback...

  写入触发：
  当 VocabularyCard 的 meaningCheckCompleted = true 或
  userSentence 不为空时，同步写入 word_memory 表。
  （upsert：如果该用户该单词已存在，则更新；否则新建）

四、间隔重复算法（SRS）设计
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

采用经典 SM-2 算法（Anki 同款），简化版：

【输入】
  q = 本次复习质量分（0~5）
      5 = 完美记忆
      4 = 正确但需要思考
      3 = 答对但有一定困难
      2 = 答错，但想起来了
      1 = 答错，几乎不记得
      0 = 完全不记得

【计算规则】
  IF q >= 3（答对）:
    IF review_count == 1: interval = 1 天
    IF review_count == 2: interval = 6 天
    ELSE: interval = round(prev_interval × ease_factor)
    ease_factor = ease_factor + (0.1 - (5-q) × (0.08 + (5-q) × 0.02))
    ease_factor = max(1.3, ease_factor)   // 最低 1.3，防止间隔收缩过快
    mastery_status:
      review_count >= 3 且 ease_factor >= 2.0 → REVIEWING
      review_count >= 5 且 correct_count/review_count >= 0.8 → MASTERED

  IF q < 3（答错）:
    interval = 1 天（重新开始间隔）
    incorrect_count += 1
    mastery_status 退回 LEARNING（若为 MASTERED 则退为 REVIEWING）

【next_review_at 计算】
  next_review_at = NOW() + interval_days × 24h

【状态机转换】
  LEARNING ──(q>=3, count>=3)──► REVIEWING
  REVIEWING ──(q>=3, count>=5, acc>=80%)──► MASTERED
  MASTERED ──(q<3)──► REVIEWING
  REVIEWING ──(q<3, 连续2次)──► LEARNING

【5.1 WordMemoryService 接口】

  interface WordMemoryService {

    // 从 VocabularyCard 同步写入/更新单词记忆（集成触发点）
    void syncFromVocabularyCard(Long userId, VocabularyCard card);

    // 获取用户所有单词记忆（分页，可按状态/时间排序）
    Page<WordMemoryDTO> getWordHistory(Long userId, WordMemoryQuery query);

    // 获取今日到期需复习的单词列表
    List<WordMemoryDTO> getDueReviews(Long userId);

    // 提交一次复习结果，更新 SRS 参数
    WordMemoryDTO submitReview(Long userId, Long wordMemoryId, ReviewSubmitRequest request);

    // 获取统计摘要（掌握总数、今日学习、连续打卡天数等）
    WordMemoryStatsDTO getStats(Long userId);

    // 给 ContextManagerService 用：生成 AI 上下文摘要字符串
    String buildVocabularyContextSummary(Long userId);
  }


【5.2 syncFromVocabularyCard 逻辑】

  当 VocabularyCard 触发任一事件时调用：
    事件A：meaningCheckCompleted = true（用户猜词义）
    事件B：userSentence 被保存（用户造句）

  逻辑：
  1. 以 (userId, word) 查询 word_memory
  2. 如果不存在 → INSERT（first_seen_at=NOW, mastery_status=LEARNING）
  3. 如果存在 → 更新：
       - 合并 source_conversation_ids（追加当前 conversationId）
       - 如果有造句，追加到 user_sentences（最多保留最新 3 条）
       - 如果 meaningIsCorrect=true，调用 SRS 引擎以 q=4 更新
       - 如果 meaningIsCorrect=false，以 q=1 更新


【5.3 buildVocabularyContextSummary 逻辑】
  （供 ContextManagerService 调用，注入系统 Prompt）

  生成格式示例：
  ──────────────────────────────
  [用户词汇记忆摘要]
  已掌握单词（MASTERED）：ambiguous, eloquent, persist, ...（最多20个）
  学习中（REVIEWING）：ephemeral, verbose, concise, ...（最多10个）
  今日待复习：scrutiny, benevolent（2个）
  ──────────────────────────────

  注意：仅在词汇扩展/对话模式的 system prompt 中注入，避免污染其他模式。
  字数控制在 200 token 以内。


【5.4 Redis 缓存策略】

  Key: word_memory:due:{userId}
  Value: 今日待复习单词 ID 列表（JSON）
  TTL: 到当天 23:59:59（每天刷新一次）
  失效条件: 用户完成任意一次复习后，清除该 key，下次请求重新计算。

  Key: word_memory:stats:{userId}
  Value: 统计摘要 JSON（总词数、各状态计数、连续打卡天数）
  TTL: 1 小时（轻量缓存，每次学习后 evict）

【7.2 UI 入口设计】

  侧边栏或顶部导航增加「词汇本」入口，点击打开 WordbookPanel。
  侧边栏「今日 N 个待复习」红点提示（从 /stats 接口读取 dueReviewCount）。
  词汇复习按钮放在 LearningModeSelector 旁边，一键进入 SRS 复习模式。


【7.3 SRS 复习交互流程】

  1. 展示单词正面（只显示 word + phonetic）
  2. 用户回想词义，点击"查看答案"翻转
  3. 展示完整卡片（meaning + example + synonyms）
  4. 用户自评：[完全不记得] [有点模糊] [想起来了] [轻松记住] [完美]
     对应 q 值：      0            1          3           4         5
  5. 调用 POST /review/{id} 提交，前端乐观更新展示下一张
  6. 复习完毕展示本次统计（复习了N个词，答对M个，下次复习最近的词等N天后）

八、AI 上下文注入设计（Memory → AI）
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【目的】
  让 AI 知道用户的词汇储备，从而：
  ① 在造句练习中使用已学词汇，加深印象
  ② 避免反复教已掌握的单词
  ③ 在推荐新词时考虑难度梯度

【注入方式】
  在 ContextManagerService.buildSystemPrompt() 中，
  对"词汇扩展"和"日常对话"模式，调用：
    wordMemoryService.buildVocabularyContextSummary(userId)
  将返回的摘要字符串追加到 system prompt 末尾。

【注入内容格式】（约 150~200 token）
  ─────────────────────────────────────────────
  [Learner Vocabulary Profile]
  Mastered (avoid re-teaching): ambiguous, persist, eloquent, scrutiny...
  Currently learning (reinforce naturally): ephemeral, verbose, benevolent
  Today's new words: concise, rhetoric
  ─────────────────────────────────────────────

【注意事项】
  - 只注入 MASTERED 和 REVIEWING 状态的单词（不超过 30 个）
  - 按 ease_factor 排序取前 N 个（最稳固的优先）
  - 在 /chat 普通聊天模式下注入；在语法练习模式下不注入，避免干扰

【场景A：学习新单词（词汇扩展对话中）】

  用户猜词义 → VocabularyCardService.saveMeaningCheck()
                │
                └─► WordMemoryService.syncFromVocabularyCard()
                      ├── UPSERT word_memory（首次 → INSERT）
                      ├── 记录 word_review_log（type=MEANING_GUESS）
                      └── 更新 SRS 参数（q=4 答对 / q=1 答错）
                            └── evict Redis cache（该用户 due list + stats）

  用户造句 → VocabularyCardService.saveUserSentence()
               │
               └─► WordMemoryService.syncFromVocabularyCard()
                     └── 追加 user_sentences JSON


【场景B：SRS 复习（独立复习模式）】

  用户点击「今日复习」
    → GET /review/due（先查 Redis，miss 则查 DB next_review_at <= NOW）
    → 前端展示闪卡序列
    → 用户完成每张卡，POST /review/{id} with qualityScore
    → WordMemoryService.submitReview()
         ├── 调用 SRS 引擎更新 interval, ease_factor, next_review_at
         ├── 更新 mastery_status
         ├── INSERT word_review_log
         └── evict Redis


【场景C：AI 对话前上下文构建】

  用户发消息 → ChatService → ContextManagerService.buildSystemPrompt()
                                  │
                                  └─► WordMemoryService.buildVocabularyContextSummary()
                                        ├── 查 Redis word_memory:stats:{userId}
                                        ├── miss → 查 DB 构建摘要
                                        └── 返回摘要字符串注入 system prompt

十一、关键设计决策与 Trade-off
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

【决策1：单词标准化 vs 保留原词】
  选择：存储时统一 toLowerCase() + trim()，phonetic 保留原样。
  原因：同一单词在不同卡片中可能大小写不同（"Eloquent" vs "eloquent"），
        标准化保证 UNIQUE(user_id, word) 不重复。
  Trade-off：不区分词性（run 作动词 vs run 作名词），接受该简化，
             因为语言学习初级阶段按单词原形记忆即可。

【决策2：同步写入 vs 异步写入 word_memory】
  选择：异步（@Async + Spring Event），不影响词汇卡主流程延迟。
  原因：syncFromVocabularyCard 涉及 UPSERT + SRS 计算，
        不应阻塞前端的即时反馈响应。
  Trade-off：极端情况下（JVM 崩溃）可能丢失最近一次同步，
             但该数据是可重建的（VocabularyCard 原始数据还在），可接受。

【决策3：SRS 复习与对话中学习是否合并】
  选择：分离。SRS 是独立的复习界面，词汇扩展对话是学习入口。
  原因：学习新词（需要 AI 引导和语境）和复习旧词（高效快速过单词）
        是不同的认知场景，强行合并会互相干扰。
  Trade-off：用户需要在两个入口之间切换，通过侧边栏红点降低摩擦。

【决策4：AI 上下文注入的字数控制】
  选择：最多 30 个单词，约 150~200 token。
  原因：注入过多会挤占有效对话上下文（模型上下文窗口有限），
        且 AI 处理超长词表时实际效果递减。
  Trade-off：仅注入最"稳固"的 MASTERED 词和 REVIEWING 中的词，
             LEARNING 阶段词汇太新，AI 提及可能造成混乱。
